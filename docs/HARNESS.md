# Harness Engineering Standards

This document defines the repo-local harness contract for Signal. It translates the relevant OpenAI Codex and eval guidance into rules this repository can enforce with Markdown, scripts, and verification commands.

Verified against official OpenAI docs on 2026-05-16:

- Codex best practices: https://developers.openai.com/codex/learn/best-practices
- AGENTS.md guidance: https://developers.openai.com/codex/guides/agents-md
- Agent skills: https://developers.openai.com/codex/skills
- Testing agent skills with evals: https://developers.openai.com/blog/eval-skills

## Harness Contract

For this repo, the harness is the full quality contract around the app and future agent workflows:

- instructions that tell agents how to work
- docs that explain architecture, scoring, testing, and workflows
- deterministic scripts that check repo knowledge and scoring behavior
- verification commands that prove a change
- captured evidence that explains failures and regressions

Keep the harness practical. Add checks when they prevent repeated mistakes, not because every preference needs a script.

## Markdown File System

- `AGENTS.md` is the short map. It should point to the docs, not contain the manual.
- `README.md` is for humans starting the repo.
- `docs/ARCHITECTURE.md` owns system shape, boundaries, and data flow.
- `docs/QUALITY.md` owns engineering taste and review rules.
- `docs/TESTING.md` owns verification commands and evidence expectations.
- `docs/WORKFLOWS.md` owns repeatable local procedures.
- `docs/HARNESS.md` owns harness standards and future eval rules.
- `docs/signal-scoring.md` owns scoring semantics.
- `docs/generated/repo-map.md` is generated by `npm run harness:update-map`.
- `scripts/harness/*` owns deterministic repo checks.

Do not duplicate detailed rules across files. Link to the owning document and update the harness check when a doc becomes required.

## Check Design

Follow this order when adding quality gates:

1. Define the objective in measurable terms.
2. Pick a small representative data set or fixture set.
3. Add deterministic checks first.
4. Keep early checks warning-first when the rule is useful but not mature.
5. Promote a warning into a failure only when the signal is stable and actionable.
6. Add slower runtime or browser checks only when they reduce real release risk.

The current local gate is:

```powershell
npm run harness
```

It checks required docs, `AGENTS.md` links, generated repo-map freshness, and large source-file visibility.

## Eval Standard

When this repo grows Codex skill or agent-workflow evals, use this structure:

- `evals/*.prompts.csv` for small targeted prompt sets.
- `evals/*.schema.json` for structured rubric output.
- `.tmp/evals/` for JSONL traces, screenshots, generated repos, and other run artifacts.
- `scripts/harness/*` for deterministic parsers and scoring checks.

Start with 10 to 20 high-signal examples. Include explicit invocations, realistic implicit prompts, edge cases, and negative controls. Grow the set from real failures.

Eval checks should cover:

- outcome: the task completed and the expected artifact exists
- process: the expected tool, command, or workflow happened
- style: output follows repo conventions
- efficiency: the run did not loop, thrash, or create noisy artifacts

Prefer deterministic checks over model-graded checks. Use a structured rubric only for qualitative rules that cannot be reliably checked with file existence, command events, HTTP responses, or TypeScript tests.

## Trace And Artifact Rules

For automated Codex evals, capture JSONL traces with `codex exec --json` so the harness can score what happened, not just the final text. Store raw traces and generated artifacts under `.tmp/` so they stay out of git unless a curated fixture is intentionally promoted.

If a trace exposes a repeated failure, turn it into one of:

- a deterministic harness check
- a scoring regression case
- a documented workflow rule
- a repo-scoped skill with a checkable definition of done

## Verification Tiers

Use the smallest tier that proves the change:

- Docs or harness metadata: `npm run harness`
- TypeScript or shared contracts: `npm run lint`, `npm run typecheck`, `npm run harness`
- Routes, dependencies, build config, or deployment behavior: add `npm run build`
- UI behavior: add browser checks from `docs/TESTING.md`
- Signal scoring: add `npm run test:scoring`

When verification fails, fix the cause and rerun the command that failed before claiming completion.

## PowerShell Harness Rules

- Use `$ErrorActionPreference = "Stop"` in harness scripts.
- Check native command exit codes when a script shells out to tools that may fail without throwing.
- Keep scripts Windows-friendly because this repo is primarily worked through PowerShell.
- Keep output short and actionable.
- Avoid new dependencies unless the existing toolchain cannot express the check clearly.

## Completion Evidence

Every substantial change should end with:

- changed files
- simplifications or new checks made
- verification commands run
- remaining risks or skipped checks

If no risks remain, say that directly.
